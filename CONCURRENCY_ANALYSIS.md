# Campus Paw-Track (校园爪迹) 并发与性能分析文档

本篇文档对 Campus Paw-Track 项目在多人同时在线、登录及进行高频操作等并发场景下的系统表现进行深度剖析，并针对潜在瓶颈给出工业级的生产环境优化方案。

---

## 一、 系统当前并发能力评估

在目前的设计下，项目运行在单台服务器上时，对于常规的**百人级同时在线、十人级并发写入**（如日常打卡、发表留言、查看地图等交互），系统能够**稳定流畅运行**。这得益于以下几个底层的优良设计：

1. **线程安全的管理员 Token 存储**
   * 后端 `AdminAuthService` 使用了 `ConcurrentHashMap` (`tokenStore`) 统一存放已登录管理员的令牌。
   * `ConcurrentHashMap` 采用分段锁（JDK 8+ 采用 CAS + synchronized 细粒度锁）机制，在多线程并发读写 Token 时**是绝对线程安全的**，避免了多线程冲突引起的崩溃。

2. **高性能数据库连接池 (HikariCP)**
   * Spring Boot 默认集成了目前性能最佳的数据库连接池 **HikariCP**。
   * 多人同时执行地图渲染（全量足迹查询）、留言发表或进行评分时，数据库连接能被迅速复用，无需频繁建立和销毁 TCP 连接，极大保护了 MySQL 数据库。

3. **安全的文件重命名（避免写冲突）**
   * `UploadController` 在接收用户上传的图片时，采用 `UUID.randomUUID().toString() + ext` 进行重命名。
   * 即使多人同一毫秒上传相同文件名的图片，落盘时也不会发生文件覆盖冲突。

4. **数据库写入与 AI 任务的异步协调**
   * 引入了 `TransactionSynchronizationManager` 提交后钩子，保证了数据库事务提交后才去触发外部 AI 异步写入任务，避免了数据库“未提交读”导致的并发竞态问题。

## 二、 潜在的高并发性能瓶颈

当系统推广至大流量场景，或面临数十万级注册用户、瞬时上千人同时操作时，以下几个环节会成为系统的核心瓶颈：

### 1. 外部 AI 接口同步阻塞（最大性能瓶颈）
* **瓶颈表现**：
  在 `UploadController` 中，用户上传图片时会同步调用阿里云通义千问模型（Qwen-VL-Max）进行“图片智能验证”和“视觉特征匹配”。
* **隐患原因**：
  AI 视觉模型处理一张图片通常需要 `1 ~ 5 秒`，期间对应的 Tomcat 请求线程会被一直挂起挂机等待。
* **高并发后果**：
  Tomcat 默认最大线程数通常为 200 个。如果瞬间有超过 200 人同时点击上传图片，Tomcat 线程池将被直接打满，后续的所有常规 Web 请求（如刷新主页、登录等）都会因为拿不到处理线程而出现排队甚至连接超时。

### 2. 内存 Token 存储的水平扩展局限（多机部署受阻）
* **瓶颈表现**：
  管理员 Token 存储在应用实例内存的 Map 中。
* **高并发后果**：
  如果为了应对高并发而采用集群部署（通过 Nginx 负载均衡），用户请求会被分发到不同的服务器节点。由于各节点的 JVM 内存是不共享的，管理员在 A 节点登录后，下一次请求若被分发到 B 节点，就会因为找不到 Token 而被迫重新登录。

### 3. 游客生成时的昵称并发重复风险
* **瓶颈表现**：
  `UserService.getOrCreateGuest` 依赖查询数据库 `userRepository.countGuests()` 来获取当前游客总数，并生成类似 `游客100` 的昵称。
* **高并发后果**：
  如果有多个新设备在极短的同一毫秒内首次进入网页，可能会同时读取到相同的数据库 `count`，从而生成相同的昵称（例如生成两个“游客100”）。虽然由于以设备 ID 作为唯一依据，这不影响系统业务逻辑，但会导致昵称出现重复。

## 三、 生产环境下的并发优化解决方案

为支持未来更大的用户体量，建议采用以下架构优化手段：

### 1. 引入异步任务队列解耦 AI 交互（推荐首选）
将“图片保存”与“AI 分析推理”解耦：
1. **同步轻量化**：用户上传图片后，后端保存文件并生成一条“处理中”状态的草稿足迹记录，直接向前端返回 `上传成功，AI 正在识别中`（耗时仅需数十毫秒），立刻释放 Web 线程。
2. **异步队列处理**：将图片分析任务放入**线程池**或**消息队列（如 RabbitMQ/Redis Queue）**中，由后台工作线程慢慢去调用外部 AI。
3. **实时推送**：AI 分析完成后，通过 **WebSocket** 实时推送通知前端更新，或由前端通过轻量级短轮询获取状态。
* **效果**：Web 容器线程不会被 AI 接口阻塞，并发吞吐量可提升 10 倍以上。

### 2. 升级为 JWT（无状态 Token）
* 将管理员的 Token 机制升级为 **JWT (JSON Web Token)**。
* 服务器不需要在本地内存中保存 Token，只需在客户端携带 Token 时进行解密签名校验。
* **效果**：后端服务实现完全无状态化，支持无缝多机集群部署。

### 3. 使用 Redis 缓存共享状态
* 如果保留 Session 机制，可以将 Token 存入 **Redis** 缓存，而不是内存的 Map 中。
* **效果**：不仅支持分布式共享，还可以设置 Token 的过期和自动清理机制。

### 4. 使用 Redis 自增生成唯一序列
* 针对游客昵称生成，使用 Redis 的 `INCR` 命令来获取唯一的递增序列号，替换传统的数据库 `count` 查询。
* **效果**：在高并发下依然能保证每个游客昵称绝对唯一。
